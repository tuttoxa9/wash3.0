# План внедрения услуги "Оклейка" (Wrapping)

Привет! Я изучил текущую структуру проекта `wash3.0` и требования. Внедрение разделенной услуги (продажа в один день, исполнение в другой с разными процентами) вполне укладывается в текущую архитектуру, но потребует аккуратной работы с финансами, чтобы деньги не задвоились в кассе.

Вот как я вижу реализацию этого функционала по шагам:

## 1. База данных (Supabase)
Мы можем пойти двумя путями, но самый надежный — добавить новую сущность для отслеживания "Ожидающих оклеек".

**Вариант А (Новая таблица `pending_wraps`):**
Создаем в Supabase таблицу для проданных, но не выполненных оклеек:
- `id`: UUID
- `sale_record_id`: ID записи продажи
- `date_sold`: Дата продажи
- `car_info`: Информация об авто
- `service`: Название услуги
- `price`: Стоимость
- `status`: `pending` | `completed`

**Вариант Б (Без новых таблиц - рекомендуемый для простоты):**
Использовать таблицу `car_wash_records`. Продажа сохраняется как `wrap_sale`. При исполнении создается запись `wrap_execution` с привязкой к оригинальной записи. Виджет "Ожидающие оклейки" будет просто искать все `wrap_sale`, для которых нет соответствующего `wrap_execution` в базе.

**Настройки (Settings):**
В глобальные настройки (`MinimumPaymentSettings`) нужно добавить два новых поля:
- `adminWrapSalePercentage`: % администратору за продажу оклейки
- `washerWrapExecutionPercentage`: % сотруднику(ам) за выполнение оклейки

## 2. Обновление типов (`src/lib/types/index.ts`)
```typescript
// Добавляем новые типы услуг
export interface CarWashRecord {
  // ...
  serviceType?: "wash" | "dryclean" | "wrap_sale" | "wrap_execution";
}

// В метод оплаты можно добавить тип "prepaid" (предоплата), 
// чтобы при исполнении оклейки деньги не прибавлялись к кассе повторно.
```

## 3. Этап Продажи (Оклейка - продажа)
* **Интерфейс:** При выборе услуги "Оклейка" блок выбора сотрудников (кто мыл) скрывается. Вместо него появляется селектор "Кто продал" (автоматически выбирается админ текущей смены, но можно поменять).
* **Логика:** Сохраняется запись типа `wrap_sale`.
* **Финансы:** Деньги идут в общую кассу (наличные/терминал) текущего дня. Выбранному "продавцу" начисляется `adminWrapSalePercentage` в зарплату за день.
* **Результат:** Запись появляется в общем списке как продажа, и автоматически попадает в новый виджет "Ожидающие оклейки".

## 4. Виджет "Ожидающие оклейки"
По аналогии с "Активными долгами", на главном экране админа будет висеть список машин, за которые взяли деньги, но не оклеили.
* На каждой карточке кнопка "Завершить/Исполнить".
* При нажатии открывается модалка: нужно выбрать дату завершения (по умолчанию сегодня) и выбрать сотрудников (мойщиков/клейщиков), которые выполнили работу.

## 5. Этап Исполнения (Оклейка - исполнение)
* **Логика:** Создается новая запись типа `wrap_execution` в день исполнения.
* **Финансы:** Важный момент — у этой записи будет цена, но тип оплаты будет техническим (например, `prepaid` или скрытый безнал), чтобы эти деньги **не шли в кассу** текущего дня (ведь они были внесены в день продажи). 
* **Зарплата:** Выбранным исполнителям начисляется `washerWrapExecutionPercentage` от стоимости. Сумма делится на количество исполнителей (если клеили вдвоем).
* **Результат:** Запись пропадает из виджета "Ожидающие", и исполнители получают свою зарплату.

## 6. Движок отчетов и зарплат (`recalculateReportTotals`)
Это самая ответственная часть. Придется доработать логику подсчета:
- Выручка дня: включает `wash`, `dryclean`, `wrap_sale`, но игнорирует `wrap_execution`.
- Зарплаты админов: процент считается от машин + процент от `wrap_sale`.
- Зарплаты мойщиков: процент считается от их моек/химчисток + процент от их `wrap_execution`.

---

**Резюме:**
Такой подход позволит полностью разделить финансовые потоки продажи и исполнения, корректно считать ЗП на разных сменах и не задвоить выручку.
