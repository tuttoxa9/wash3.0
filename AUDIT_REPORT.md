# Отчёт по аудиту проекта "DetailLab"

## Введение
Был проведен полный аудит архитектуры, бизнес-логики и компонентов приложения DetailLab. Основной упор был сделан на корректность расчёта заработных плат, работу с историческими данными (сменами) и структуру компонентов. В целом проект имеет хорошую структуру и использует современные практики, однако обнаружены критические ошибки в логике расчётов в некоторых местах, а также дублирование кода, которые могут привести к неверным финансовым отчётам и путанице при поддержке.

Ниже представлен детальный разбор проблем с распределением по уровню риска.

---

## 🔴 Уровень риска: Критический

### 1. Хардкод процентов зарплат в компонентах
**Проблема:**
В файле `HomePage.tsx` в функции `getEmployeeStats` (строка ~130) и компоненте `EmployeeDetailModal` (строка ~3500) проценты заработка сотрудников захардкожены:
```javascript
if (role === "washer") {
  percentage = isDryClean ? 40 : 30; // Хардкод!
} else if (role === "admin") {
  percentage = isDryClean ? 20 : 15; // Хардкод!
}
```
**Риск:**
Даже если администратор изменит проценты выплат в настройках приложения (`minimumPaymentSettings`), на "Главной странице" (HomePage) в карточках сотрудников и в детальной модалке работника будут отображаться старые (захардкоженные) цифры. Это вызовет конфликт между реальными расчетами (которые делаются в `SalaryCalculator`) и тем, что видит пользователь на экране.

**Решение:**
Удалить хардкод. Использовать централизованную утилиту `calculateEmployeeShare` из `src/lib/employee-utils.ts`, передавая в неё актуальные настройки `state.minimumPaymentSettings`.

### 2. Использование текущих ролей вместо исторических
**Проблема:**
В `HomePage.tsx` при расчете заработка сотрудника в `getEmployeeStats` и в `EmployeeDetailModal` роль сотрудника берется из его текущего профиля:
```javascript
const role = state.employees.find(e => e.id === employeeId)?.role || 'washer';
```
**Риск:**
Если сегодня мойщика повысят до администратора, то при просмотре вчерашней смены приложение посчитает ему зарплату по ставке администратора, хотя вчера он был мойщиком. Это нарушает целостность финансовых данных.

**Решение:**
Использовать исторические роли, сохраненные для конкретной смены. Они доступны в `currentReport.dailyEmployeeRoles` или через сервис `dailyRolesService`. Утилита `determineEmployeeRole` уже существует для правильного определения исторической роли — её необходимо внедрить везде, где идет расчет ЗП.

## 🟡 Уровень риска: Средний

### 3. Дублирование кода компонентов и синтаксическая ошибка в экспорте
**Проблема:**
Огромные модальные окна (`DailyReportModal`, `AddCarWashModal`, `CloseDebtModal`) полностью скопированы и вставлены прямо в файл `HomePage.tsx`, занимая более 2000 строк кода.
При этом в папке `src/components/Home/` существуют отдельные файлы с этими же компонентами. Более того, в конце файла `src/components/Home/DailyReportModal.tsx` присутствует критическая синтаксическая ошибка двойного дефолтного экспорта:
```javascript
export default HomePage;
export default DailyReportModal;
```

**Риск:**
Изменение логики в одном месте (например, в инлайн-модалке в `HomePage.tsx`) не применится к отдельному файлу. Это делает поддержку проекта мучительной. Ошибка экспорта может привести к падению приложения при попытке импорта компонента.

**Решение:**
Очистить `HomePage.tsx` от инлайн-компонентов. Импортировать все модалки из `src/components/Home/`. Удалить строку `export default HomePage;` из файла `DailyReportModal.tsx`.

### 4. Терминологическая путаница ("Доля" vs "Заработок")
**Проблема:**
В таблице `EmployeeDetailModal` колонка называется "Доля", но значения в ней — это итоговый заработок сотрудника с одной машины (доля от выручки * процент сотрудника). В отчетах Word под "Долей" подразумевается стоимость услуги, разделенная на количество участников.

**Риск:**
Путаница при аудите финансов. Сотрудники могут не понимать, как получилась итоговая цифра.

**Решение:**
Переименовать колонку "Доля" в "Заработок" в UI, чтобы соответствовать логике.

---

## 🟢 Уровень риска: Низкий

### 5. Лишние/неиспользуемые файлы
**Проблема:**
В корне проекта есть файлы-скрипты `patch_salary.sh`, `patch_settings.sh`, `patch_ui_tabs.sh`, `fix-loading.js`, `test.js`, `out.txt`.

**Риск:**
Они создают визуальный мусор и могут быть случайно запущены на боевом сервере.

**Решение:**
Удалить эти артефакты, если они больше не нужны для разработки.

---

## Итог и План действий
В целом, приложение написано аккуратно, логика расчетов в ядре (SalaryCalculator) правильная. Проблемы кроются исключительно в UI-представлении (на Главной странице), где разработчик использовал заглушки (хардкод) вместо вызовов функций ядра.

**Предлагаю следующие шаги для немедленного исправления:**
1. Починить `DailyReportModal.tsx` (удалить дублирующийся экспорт).
2. Удалить простыни инлайн-кода модалок из `HomePage.tsx` и заменить их импортами.
3. Прокинуть исторические `employeeRoles` и `minimumPaymentSettings` во все модалки.
4. Заменить хардкод `isDryClean ? 40 : 30` на вызов `calculateEmployeeShare(...)`.
